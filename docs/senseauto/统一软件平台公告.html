统一软件平台公告

统一软件平台

这个流程是经过多次与专业组leader和多数核心开发讨论得出的，涉及后续绝大多数同学的工作流程，请各位重点关注。

切到统一软件平台后，研发同学将主要在develop分支上开发，根据规划，不定期往master提出合入请求和发版测试。一般情况下，不直接往master分支直接提交代码。

develop分支的合入权限已经分发给各个专业组sdk负责人，开发者的topic经过质量迭代体系的自测和组内review后，可以由sdk负责人合入develop分支。

研发流程图：

1. 发版规则

1.1 版本形式

统一软件平台版本发布按照deb包的形式发布，每个模块一个deb包，整个senseauto由若干个deb包组成。

1.2 版本号规则

  • sdk：各模块按照各自的需求发版，合入master，版本号自己确定，总体规则按照：Major.Minor.Patch 来。完成版本合入之后打上tag。
  • senseauto：会有一个整体的版本，初步按照一个季度一个版本。senseauto版本是由各个模块的版本组合而成，两者的版本号不相关。

1.3 发版提测时机

  • 定期每周一开始发版测试，专业组sdk如果能赶上则提交版本测试，不能赶上则下周一再提交。
  • 如果多个专业组同时提测，测试组可以一并测试。
  • 如果模块间改动有依赖，则必须一起进行发版测试和发版。
  • 重大修改，可以单独提测发版。

1.4 接口变动如何发版

接口变动类型包括：ros message、framework sdk、公共config。

接口变动，只能系统组合入，确保系统正常运行。

如果由接口变动导致的多个模块需要同时修改，则有如下注意点：

  • 联合topic必须基于master分支进行开发与合入
  • 联合topic必须同时合入master并发deb版本

合入方式包括：

  • 联合topic在周一发版测试中一起测试
  • 如果联合topic改动非常大，也可以对联合topic单独做发版测试（需提前向测试组提出需求，好计划测试排期）

1.5 多个项目发版支持

专业组只维护一个master分支来支持不同项目，如L4和L2+，但是针对不同项目进行不同的发版与测试。

具体来说就是：如果L4项目的功能点达到预期，则从develop向master提出发版请求，经过了L4的发版测试项目之后，则可以合入master分支，并打上tag并发布L4的deb包；L2+发版类似。

项目组需要保证develop的稳定支持L4和L2+的项目。

如下图所示，ad v3.0.0发版后只需经过L4项目的正规发版流程，才能合入并发布版本；adas L2+ v1.0.0版本，只需经过L2+的发版流程即可。

2.发版测试

2.1 发版测试流程

  • 版本提测

    （

    1 专业组SDK自测

    ）

    1. 提测前2天（周四前给出）给出release note，方便测试组提前制定测试计划。release note包括：
      1. 本次发版的所有的新feature列表
      2. 本次发版修复的Jira issue列表（来自于之前的日常master测试测出的问题）
    2. 专业组提供自测报告，包括（仿真、离线测试、路测mpi mphb等指标），自测ok之后才能进入发版测试环节
    3. 提交develop到master的topic

  • 集成测试，按照制定的测试计划进行（2 集成测试

    ）

    • 自动化测试：ci、

      仿真、

      评测

      • 评估自动化测试指标

    • 离线测试

    • 路测

      • 路测组与专业组商量路测里程、路测通过标准等

  • 测试问题反馈

    • 测试过程中确保尽早暴露问题，当天测试、当天分析、当天反馈，提升开发和测试效率
    • 测试过程所有问题以Jira issue进行反馈，归属于发版测试类别（由专业组跟踪这些issue的完整生命周期）
    • 测试期间发现小问题，专业组如评估需修复，则需尽快hotfix patch修复并重新发版，然后继续测试。
    • 如果某个模块问题过多或者严重问题，则退回该模块sdk的版本，其他模块继续测试

  • 通过标准

    • 测试通过的硬性指标：
      • 仿真case通过率不能下降，不能有新增失败case
      • mpi和mphb稳定500KM里程以上，与master上4周的版本指标标准差的2倍以内，并且没有新问题出现。
    • 满足release note要求功能
      • 新feature list需要全部构造场景通过
      • Jira issue bug list全部进行回归测试通过，包括仿真或者路测。

  • 测试通过与发版

    • 测试通过后，gerrit 合并代码，发布deb版本。
    • 开发者和测试人员自动更新到最新版本进行测试。
    • master日常测试也更新到新版本进行路测。

  • 复盘

    • 测试组主导发版报告会议的讨论
    • 形成发版测试报告
    • 复盘开发和测试过程，总结经验，形成文档，优化流程

备注：

  • 发版测试期间，提测模块的develop可以继续合入
  • 禁止对多个模块并行进行发版测试，合入发版有先后，后提测的版本必须等先提测的完成版本合入之后，再基于新版本进行测试。

3.master日常测试

3.1 测试方式

  • 固定3辆车每天使用当天最新master版本进行测试

3.2 测试问题修复方式

  • master测试出的频繁出现的小问题，也可以以hotfix的形式修复，版本号以patch num形式追加（或者看具体情况是否需要hotfix，不建议经常使用hotfix的形式修复master的问题）
  • 不以hotfix形式修复的测试问题，反馈给各组在develop上修复，然后从develop中提测合入master。

3.3 规范的jira issue

  • master测试发现的问题，在专业组确认后，由测试组创建issue
  • Jira issue的整个生命周期由测试组管理，包括创建、回归测试、关闭 。禁止由专业组自行关闭master测试issue。
  • 每个issue确认或解决之后，都应该给出issue产生原因，比如编码bug、未考虑到的场景、明确不支持场景

3.4 规范的case派发

  • 当天测出的case需要按时推进完成模块划分（第二天上午分前一天的所有case）
  • 由测试组负责每个case的完整生命周期，包括上传、分析、派发、确认
  • 测试组确保每个被专业组确认的case都需要绑定到对应的某个issue上，方便统计MPI、MPHB水平（如是bug？不支持场景？）

4.发版测试中开发与测试职责划分

开发和测试要有清晰的职责和相关评估指标

4.1 开发的职责

  • 确保自测完善，以不被退回提测作为开发团队的目标
  • 不被退回的准则：release note中的feature等都需要通过、相关指标都达到，可能会经过几次修复。
  • 大问题直接退回，大问题的定义：节点崩溃、性能明显下降影响使用，MPI明显不足。

评估指标：提测成功率

4.2 测试的职责

这里给出一些宏观上的要求：

  • 测试过程中确保尽早暴露问题，当天测试、当天分析、当天反馈，提升开发和测试效率
  • 以杜绝重大bug未测出导致合入master中为目标

评估指标：维护老问题list或case库，出现的新问题都算测试组的责任。
